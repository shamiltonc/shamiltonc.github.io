Hello! My name is Samantha Hamilton and this is my code review. Per my ePortfolio proposal in Module 1, I am reviewing a project from one of last terms’ classes- a RESTful API for CS-340 Client/Server Development. This RESTful API is split into three files- one called “FinalProject_CRUD”, one called “FinalProject_Retrieval”, and a third called “FinalProject_REST”. The purpose of these three Python files are to, respectively, perform basic CRUD operations, basic data retrieval, and process RESTful queries by use of URIs. I stand by my decision to split these up into three documents, for organization’s sake. I believe doing so keeps the RESTful API logically organized.

Comment and Coding Conventions

Two out of three files are lacking in comments. The third, “FinalProject_REST”, has proper commenting at the start of each of the declared URIs. All three follow many coding conventions, such as proper indentation (which is necessary for Python), logical flow, and proper naming conventions. My variable names are descriptive and relevant. They say, without having to add an additional comment, exactly what that variable’s purpose is, such as “myQuery” or “newCollection”.

Error Handling

This RESTful API uses HTTP errors. When running a file, a localhost server is created and each call of each function results in the output of an HTTP error such as 400, 404, or 200.

Performance

This RESTful API is easy on performance. It is not a demanding application. The code is structured in such a way that performance is optimized as much as possible.

Functionality

The files “FinalProject_CRUD” and “FinalProject_Retrieval” function completely as intended, with documentation that includes screenshots and explanation to go along with it. The file “FinalProject_REST” has the same documentation and the localhost server runs, but the functions do not execute as intended.
